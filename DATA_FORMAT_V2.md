# 租屋数据格式规范 v2.0

> **设计理念**：保留 Gemini 灵活的分析能力，将提炼信息分散到前端需要的结构化字段中

---

## 核心原则（继承自 universal_prompt.md）

### ✅ 保持不变
1. **只记录例外，不记录常态**
   - 记录：`🐶🐱可宠`（例外）
   - 不记录：`禁宠`（常态）

2. **客观性过滤**
   - 忽略：`温馨`、`舒适`、`机能齐全`
   - 保留：`全新装潢`、`步行5分钟到捷运`

3. **信息模糊警示**
   - 楼层不明 → `summary_notes: ["资讯模糊(楼层)"]`
   - 坪数不明 → `summary_notes: ["资讯模糊(坪数)"]`

---

## 文件格式

### 文件路径
`public/data/rentals.json`

### 文件结构
```json
[
  { /* 物件1 */ },
  { /* 物件2 */ }
]
```

## 字段定义与映射规则

### 1. 顶层必填字段

```typescript
{
  "id": string,              // 格式：YYYY-MM-DD-XXX
  "district": string,        // XX市XX区（如：台北市南港區）
  "housing_type": string,    // 四选一：雅房|分租套房|獨立套房|雅套混合
  "rent_price": number,      // 月租金（数字）
  "title": string,           // 物件标题（提炼关键卖点）
  "summary_notes": string[], // 警告/重要提醒（数组）
  "details": { ... },        // 详细信息对象
  "post_date": string,       // YYYY-MM-DD
  "source_url": string       // Facebook 原始链接
}
```

---

### 2. `housing_type` 判断规则

| 原始描述 | housing_type | 说明 |
|---------|-------------|------|
| 整层、独立卫浴、独立厨房 | `獨立套房` | |
| 分租、套房 | `分租套房` | |
| 雅房、共用卫浴 | `雅房` | |
| 同时有雅房和套房 | `雅套混合` | |
| **顶加+独立套房** | `獨立套房` | `summary_notes` 加 `顶加独租` |

---

### 3. `title` 生成规则

**目的**：用一句话概括物件最吸引人的特点

**优先级**（选最重要的 2-3 个组合）：
1. 交通优势：`步行X分钟到捷运XX站`
2. 特殊优势：`🐶🐱可宠`、`可租补`、`全新装潢`
3. 设施亮点：`有阳台`、`独立厨房`、`电梯大楼`

**示例**：
- `"距離捷運站1分鐘、🐶🐱可宠、附家具"`
- `"步行6分钟到南京三民、有电梯、有阳台"`
- `"全新装潢、可租补、独立厨房"`

**不要包含**：主观形容词（温馨、舒适等）

---

### 4. `summary_notes` 字段（保留灵活性）

**类型**：`string[]`

**用途**：存储重要警告、信息冲突、特殊说明

**常见内容**：
```javascript
[
  "资讯模糊(楼层)",
  "价格可能不含管理费",
  "标题与内文坪数不一致",
  "重复物件(2025-11-14)",
  "顶加独租",
  "注意：同地址有多笔物件, 此为第2笔"
]
```

**规则**：
- 没有警告时使用 `[]`（空数组）
- 每条简洁明了（≤ 30 字）
- 优先级：资讯模糊 > 价格警告 > 其他

---

### 5. `details` 对象结构

```typescript
{
  "address": string | null,
  "layout_info": {
    "room_size": string | null,      // "1房1衛"、"4房2衛"
    "overall_layout": string | null,  // "整層3房2廳2衛"
    "floor_info": string | null       // "5F"、"3F/7F"
  },
  "rights": string,                   // 权利说明
  "cost_notes": string[],             // 费用说明数组
  "facilities": string[],             // 设施数组
  "restrictions": string[]            // 限制条件数组
}
```

---

### 6. 字段映射规则详解

#### 6.1 `details.address`
- **规则**：有就填，没有就 `null`
- **示例**：
  - `"台北市南港区南港路一段123号"` ✅
  - `"捷运公馆站附近"` ✅
  - `"罗斯福路"` ✅
  - `null` ✅

#### 6.2 `details.layout_info.room_size`
- **格式**：`"X房X衛"` 或 `"X房X廳X衛"`
- **示例**：
  - `"1房1衛"` ✅
  - `"4房2衛"` ✅
  - `"2房1廳1衛"` ✅
  - `null`（未提供）

#### 6.3 `details.layout_info.floor_info`
- **格式**：`"XF"` 或 `"XF/YF"`
- **特殊规则**：
  - 楼层 ≥4 且无电梯 → 在 `facilities` 中注明 `"无电梯"`
  - 顶加 → `"5F顶楼加盖"`
  - 房间在特定楼层但总楼层是1-2F → `"房间在2F(总楼层1-1F)"`

**从 notes 迁移示例**：
```
旧 notes: "4F, 无电梯"
新格式:
  layout_info.floor_info: "4F"
  facilities: ["无电梯"]
```

#### 6.4 `details.rights`（⭐ 新增字段）

**允许值**：
- `"可租補"`
- `"可報稅"`
- `"可租補、可報稅"`
- `"🐶🐱可宠"`
- `"🐶可养狗"`
- `"🐱可养猫"`
- `"🐶🐱可宠(有条件)"` - 如需说明条件
- `"可租補、🐶🐱可宠"`（组合）
- `"未說明"`（默认值）

**从 notes 迁移示例**：
```
旧 notes: "🐶🐱可宠, 可租补"
新格式:
  rights: "可租補、🐶🐱可宠"
```

#### 6.5 `details.cost_notes`（从 notes 迁移）

**类型**：`string[]`

**内容**：费用相关的客观事实

**迁移规则**：
```
旧 notes: "电费$5/度, 管理费$1000/月, 租金含水、网路"
新格式:
  cost_notes: [
    "电费：$5/度",
    "管理费：$1000/月",
    "租金含水、网路"
  ]
```

**常见模式**：
- `"电费：台电计费"`
- `"电费：$5.5/度(含水、网路、第四台)"`
- `"水费：$500/月"`
- `"管理费未说明"`
- `"押金两个月"`
- `"退租清洁费$XXX"`

#### 6.6 `details.facilities`（从 notes 迁移）

**类型**：`string[]`

**迁移规则**：只记录**明确提到**的设施

```
旧 notes: "冷气, 洗衣机, 冰箱, 网路, 电梯, 阳台"
新格式:
  facilities: ["冷气", "洗衣机", "冰箱", "网路", "电梯", "阳台"]
```

**常见设施**：
- 基本：`冷气`, `洗衣机`, `冰箱`, `热水器`
- 厨房：`独立厨房`, `瓦斯`, `微波炉`, `气炸锅`, `电锅`
- 网路：`网路`, `第四台`, `电子锁`
- 建筑：`电梯`, `阳台`, `车位`, `顶楼加盖`
- 家具：`附家具`, `双人床`, `衣柜`, `书桌椅`
- 窗户：`气密窗`, `隔音对外窗`, `无采光窗户`
- 特殊：`全新装潢`, `全新洗衣`, `全新家电`, `全新冷气`, `全新卫浴`

**不记录**（常态设施）：
- 对于雅房/分租套房：`禁煮`, `可晾衣`, `共用洗衣机`（除非原文特别强调）
- WiFi（太普遍，除非特殊说明如"需自行装设"）

#### 6.7 `details.restrictions`（从 notes 迁移）

**类型**：`string[]`

**迁移规则**：只记录**明确限制**

```
旧 notes: "限女性, 不可开伙"
新格式:
  restrictions: ["限女性", "不可开伙"]
```

**常见限制**：
- 性别：`限女性`, `限男性`, `限女(室友皆为女性)`
- 身份：`限学生`, `限上班族`
- 生活：`不可开伙`, `禁止吸烟`, `每房限一人`
- 租期：`短租`（若 <1 年），`可短租`

**不记录**（太普遍）：
- `禁宠`, `禁止养宠`, `禁煮`, `可晾衣`（市场常态）

## 迁移策略（给 Gemini 的指引）

### 步骤 1：保持现有分析流程
1. 客观性过滤（忽略主观形容词）
2. 需求性评估（筛选重要信息）
3. 提炼关键卖点

### 步骤 2：信息分类
将提炼的信息按类型分配到对应字段：

| 信息类型 | 目标字段 | 示例 |
|---------|---------|------|
| 权利/宠物 | `details.rights` | `🐶🐱可宠`、`可租补` |
| 费用 | `details.cost_notes` | `电费$5/度` |
| 设施 | `details.facilities` | `冷气`、`电梯` |
| 限制 | `details.restrictions` | `限女性` |
| 楼层 | `layout_info.floor_info` | `5F` |
| 格局 | `layout_info.room_size` | `1房1衛` |
| 警告 | `summary_notes` | `资讯模糊(楼层)` |

### 步骤 3：生成 title
从分类后的信息中选择 2-3 个最重要的组合成 title。

### 步骤 4：组装 JSON
按照字段顺序输出完整 JSON。

## 数据验证规则

### 必须通过的检查：
1. ✅ `id` 唯一且符合格式 `YYYY-MM-DD-XXX`
2. ✅ `district` 包含"市"和"區"
3. ✅ `housing_type` 是四选一
4. ✅ `rent_price` 是正整数
5. ✅ `post_date` 是 `YYYY-MM-DD` 格式
6. ✅ `source_url` 是有效 Facebook URL
7. ✅ 所有数组字段（即使为空）必须是 `[]`，不能是 `null`
8. ✅ `details.rights` 不能为空字符串，至少是 `"未說明"`

## 常见问题处理

### Q1: 如何处理复杂的权利说明？
**A**: 组合多个权利，用顿号分隔
```json
"rights": "可租補、可報稅、🐶🐱可宠"
```

如果宠物有条件限制：
```json
"rights": "🐶🐱可宠(需亲人、乖巧)"
```

### Q2: 费用信息很复杂怎么办？
**A**: 每条独立，保持原有的简洁风格
```json
"cost_notes": [
  "电费：$5.5/度(含水、网路、第四台)",
  "管理费：$1200/月",
  "租金含水费",
  "退租清洁费$800"
]
```

### Q3: 设施太多怎么办？
**A**: 全部列出，前端会自动处理显示
```json
"facilities": [
  "全新装潢", "冷气", "洗衣机", "冰箱", "热水器",
  "网路", "第四台", "电梯", "阳台", "车位",
  "附家具", "独立厨房", "瓦斯"
]
```

### Q4: 室友信息放哪里？
**A**: 放在 `restrictions` 中
```json
"restrictions": [
  "限女性",
  "现有室友2女1男"
]
```

### Q5: 重复物件如何处理？
**A**:
1. 如果有核心条件变更（租金、空房数）→ 正常结构化 + 在 `summary_notes` 注明
```json
"summary_notes": ["重复物件(2025-11-14)", "A雅房租金从$12700调降至$12300"]
```

2. 如果无变更 → 不输出 JSON，直接报告"此为重复物件，无任何更新，已忽略"

## 输出格式要求

**最终输出必须是单行 JSON**（不含注释）：

```json
{"id":"2025-11-17-001","district":"台北市南港區","housing_type":"獨立套房","rent_price":18800,"title":"步行5分钟到南港展览馆站、🐶🐱可宠、全新装潢","summary_notes":[],"details":{"address":null,"layout_info":{"room_size":"1房1衛","overall_layout":null,"floor_info":"5F/6F"},"rights":"可租補、🐶🐱可宠","cost_notes":["电费：$5.5/度(含水、网路、第四台)","管理费：$1200/月"],"facilities":["全新装潢","冷气","洗衣机","冰箱","网路","电梯"],"restrictions":[]},"post_date":"2025-11-17","source_url":"https://facebook.com/..."}
```

**在 JSON 后面，必须附上一段"判断说明"**：
```
【判断说明】
- 顶加独租：根据「顶楼」关键字判断，加入 summary_notes
- 宠物权利：原文明确提到「可养猫狗」，放入 rights
- 费用整合：将电费、管理费分别列为独立项目
- 设施筛选：忽略主观描述「温馨舒适」，保留客观设施
```

## 部署流程

1. **Gemini 清洗数据** → 生成多个单行 JSON
2. **合并为数组** → 放入 `public/data/rentals.json`
3. **提交 GitHub**：
```bash
cd rental-website
git add public/data/rentals.json
git commit -m "更新租屋数据 - $(date +%Y-%m-%d)"
git push origin main
```
4. **部署**：
```bash
npm run build && npm run deploy
```
